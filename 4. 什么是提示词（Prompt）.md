# 什么是提示词（Prompt）
提示词（Prompt）是你与 AI 模型对话时输入的指令、问题或上下文，用于引导模型生成你期望的回复。它就像是对 AI 的“编程语言"——同样的模型，不同的提示词会产生截然不同的输出质量。


*示例**：你问Kimi"给我讲一个关于太空的故事"，这句话就是一个提示词，它引导Kimi生成一个关于太空的故事。

## 怎么样写好提示词？

### 技巧一：指令清晰，把模糊诉求变成可执行的指令链

- 很多时候，模型效果不好，往往是因为提示词不够“可执行”。一个好提示词（Prompt）的核心，是把模糊目标拆解成明确任务 + 约束 + 步骤，让模型按你期望的流程产出结果。
- 经典框架：CRISPE 原则，帮助你系统地组织提示词：
<img width="2996" height="1494" alt="image" src="https://github.com/user-attachments/assets/9b96a8fe-d95b-40b8-af1f-d8da129dbdde" />
  - Context（背景）：补充任务背景、使用场景及相关约束与边界条件。
  
  - Role（角色）：明确 AI 的角色设定、专业能力与视角立场，决定它如何思考与取舍。
    
  - Instruction（任务指令）：清晰描述要完成什么、交付什么结果，避免"写一下""想个办法"这类泛指表达。
    
  - Steps（执行步骤）：把复杂任务拆解为可操作的步骤，确保输出路径可控。
    
  - Parameters（参数约定）：约束输出格式、字数范围、语气风格、结构要求等。
    
  - Example（示例参考）：提供样例或参考标准，帮助模型对齐你期望的输出效果。

  
```
【Context】我们正在开发一款面向年轻人的健康管理 App。
【Role】你是一位资深产品经理，擅长从用户需求出发进行功能定义与流程设计。
【Instruction】请撰写一份产品需求文档（PRD），覆盖目标用户、核心功能与关键流程。
【Steps】
1) 分析目标用户画像与核心痛点
2) 定义核心功能模块与优先级（可用 MoSCoW）
3) 设计关键用户流程（用分步骤描述即可）
【Parameters】使用 Markdown；总字数 800 字以内；表达专业、简洁、可落地。
【Example】参考附件中的 PRD 模板（结构与颗粒度对齐即可）。
```

### 技巧二：角色提示（Role Prompting），让 Kimi 站在专业视角思考

角色提示（Role Prompting）是一种提示工程技术——通过赋予模型特定的身份或职业角色，激活其对应的知识领域与行为模式。当你在提示词中明确指定 Kimi "扮演"某个角色，如资深 Python 工程师、创意写作导师、古希腊历史学家等，Kimi 会基于该角色通常具备的专业知识、表达风格和价值取向来生成回应。
一个好的角色设定应包含三个要素：专业背景 + 能力边界 + 立场视角，对应以下三个层次：
- 基础层：职业身份，如“你是一位心理咨询师”
- 进阶层：能力特征 + 风格偏好，如“你是擅长认知行为疗法的心理咨询师，表达温和、重视实操建议”
- 高级层：立场 + 约束，如“你是站在来访者角度的心理咨询师，避免说教，不做诊断，只提供陪伴式引导”
层次越高，角色设定越精确，模型的回应也越贴近你的预期。来看一个对比示例：

```
❌ 弱角色：“你是个老师，帮我讲解这个概念。”

✅ 强角色：“你是一位有 10 年经验的高中物理老师，擅长用生活案例类比抽象概念。
现在要向基础薄弱的学生解释'动量守恒定律'，要求：避免公式堆砌，多用直观例子，语言通俗易懂。”

💡强角色示例中，职业身份、能力特征与任务约束缺一不可，三者共同构成了一个可执行的角色提示。
```

【角色提示词万能公式】
如果你不确定如何写角色提示词，可以直接把下面这段话发给 Kimi，让它帮你生成：

```
我需要你帮我生成一个角色提示词。

我的任务是：[用一句话描述你要做什么]
我的背景是：[你的身份、水平或使用场景，如"我是初学者""我是创业者"等]
我期望的输出是：[描述你希望得到什么样的回答，如"步骤清晰""语言通俗""有具体案例"等]

请按照以下结构生成角色提示词：
1. 职业身份：该角色是谁，有多少年经验，擅长什么领域
2. 能力特征：该角色具备哪些专业能力或思维方式
3. 风格偏好：该角色的表达风格、沟通方式
4. 立场约束：该角色应避免什么，优先考虑什么

直接输出可用的角色提示词，不需要解释。
```

### 技巧三：格式清晰——结构化Prompt

结构化 Prompt，将你的指令按照固定的逻辑框架拆分组织，而不是用一段话“随意描述”你的需求。
就像写一份工作简报——有主题、有背景、有要求、有交付标准，而不是走进会议室说“你帮我搞一下那个项目”。
结构化 Prompt 通常包含四个核心要素：角色、任务、背景、输出格式。
四要素框架

| 要素 | 说明 | 示例 |
|:---|:---|:---|
| 角色 | 告诉Kimi它将扮演谁 | 你是一位资深品牌策划顾问 |
| 任务 | 明确你要它做什么 | 为新消费品牌撰写一份品牌定位方案 |
| 背景 | 提供必要的上下文 | 目标用户是 25-35 岁都市女性，预算有限 |
| 格式 | 规定输出的结构和形式 | 用三段式输出，每段不超过 150 字 |

基础模版
```
# 角色
你是一位 [具体角色]，擅长 [相关领域]。

# 任务
请帮我 [具体动作] + [目标对象]。

# 背景
- [限制条件或参考信息]
- [目标受众或使用场景]

# 输出格式
请按以下结构输出：[格式要求]
```

进阶用法：XML 标签
面对复杂任务，可以用 XML 标签替代标题符号，让结构更严谨：

```
<role>你是一位资深产品经理</role>

<task>根据用户反馈，提炼核心痛点并给出优化建议</task>

<context>
产品：XX 协作工具
反馈内容：[粘贴原始反馈]
</context>

<output_format>
- Top 3 核心痛点（按优先级排序）
- 每个痛点的优化方向
- 预期改善效果
</output_format>
```

### 技巧四：明确输入和输出，减少信息不对称

明确告诉AI该做什么显性指令与隐性指令的区别在提示词设计中，指令的表达方式直接影响模型的执行准确性。
- 显性指令（Explicit Directive）
直接、无歧义地陈述任务要求的表达方式，例如：“请用200字总结以下文本的三个核心观点”，其中”总结”、“200字”、“三个核心观点”都是明确可量化的要求。

- 隐性指令（Implicit Directive）
依赖于模型的推断能力，如”处理一下这段文字”，模型需要猜测”处理”具体指总结、翻译、改写还是情感分析。
使用行动动词开头以行动动词开头的提示词能够立即建立任务框架，帮助模型快速识别所需的处理能力。
常用的行动动词包括信息处理类（总结、分析、比较、分类、提取）、内容生成类（撰写、创作、扩展、改写、翻译）和结构化输出类（列出、表格化、流程化、层次化）。

怎么做？
用行动动词开头

以行动动词开头的 Prompt 能够立即建立任务框架，帮助 Kimi 快速识别所需的处理方式。根据任务类型，常用动词可以分为三类：
信息处理类： 总结、分析、比较、分类、提取
内容生成类： 撰写、创作、扩展、改写、翻译
结构化输出类： 列出、表格化、流程化、层次化
动词选得准，Kimi 的执行方向就不会跑偏。

明确输入边界
告诉 Kimi 你提供的是什么材料，它的性质和范围是什么，避免它自行发挥或引入你没有提供的信息。

明确输出规格
从三个维度约束输出：形式（段落、列表、表格）、长度（字数、条数）、内容要求（几个观点、是否需要举例、要不要给出结论）。


案例对比
❌ 隐性指令
处理一下这段用户反馈。
Kimi 不知道你想要摘要、分类、情感判断还是改写建议，只能凭概率给你一个"最可能的答案"。

✅ 显性指令
请分析以下用户反馈，提取其中的核心问题，按"功能缺陷 / 体验问题 / 需求建议"三类分类整理，每类列出不超过3条，用一句话概括每条问题。
Kimi 清楚地知道：动作是"分析+提取+分类"，输入是"用户反馈"，输出是"三类、每类最多3条、每条一句话"。执行路径完全确定，结果高度可控。

### 技巧五：举例子（Few-shot Learning），用样本对齐预期

举例子，就是在 Prompt 里直接给 Kimi 看几个"输入→输出"的示范样本，让它通过模式识别来理解你真正想要的结果，而不是单靠文字描述去猜。

这对应两种不同的提示方式：
- 零样本提示（Zero-shot Prompting） 是指不提供任何示例，直接用指令描述任务。比如"请判断以下评论的情感倾向"。Kimi 会凭借自身的训练知识来理解并执行，适合通用性强、表达清晰的简单任务。
- 少样本提示（Few-shot Prompting） 则是在指令之外，额外嵌入 2-5 个示范样本，每个样本都包含一组"输入"和对应的"期望输出"。Kimi 通过观察这些样本，识别出你的判断标准、输出风格和格式规范，从而对齐你的预期。

为什么？
语言是模糊的，描述往往无法穷尽所有细节。你说"语气轻松一点"，Kimi 理解的"轻松"和你心里的"轻松"可能相差甚远。但如果你直接给它看一个你满意的例子，它立刻就能明白你要的感觉。
少样本提示解决的核心问题是标准传递。尤其在以下场景中，光靠指令往往不够：
输出风格高度个性化时，比如你有固定的品牌语气或写作腔调，用文字描述很难说清楚，但一个示例就能让 Kimi 快速复刻。
判断标准比较主观时，比如内容分级、情感分类、质量评估，不同人有不同尺度，示例能帮 Kimi 校准到你的标准。
格式要求比较复杂时，表格、特定排版、固定模板，示例比格式说明更直观、更不容易出错。

```
[任务说明]

示例1：
输入：[样本输入]
输出：[期望输出]

示例2：
输入：[样本输入]
输出：[期望输出]

现在请处理：
输入：[你的实际输入]
输出：
```

三个关键原则
- 示例要有代表性。 选择能覆盖典型情况的样本，而不是只挑最简单的。如果任务有边界情况，最好在示例中也体现出来。
- 输入输出要对齐。 每个示例的输出，必须是你真正满意的结果。Kimi 会把你的示例当作"标准答案"来学习，示例质量直接决定输出质量。
- 数量控制在 2-5 个。 太少不足以建立模式，太多会稀释关键信息、增加处理负担。大多数任务 3 个示例就足够。

### 技巧六：分布解决思维链(Chain of Thought)
思维链（Chain of Thought，简称 CoT）是一种引导 Kimi 把解题过程显性化的提示方式。它不是直接让 Kimi 给出答案，而是让它像人一样，一步一步把推理过程写出来，最后再得出结论。
简单说，就是让 Kimi "把思考过程说出来"，而不是只告诉你结果。
这种方式来源于一个发现：对于复杂问题，当模型被要求逐步推理时，准确率会显著高于直接输出答案。原因在于，分步推理迫使模型在每一步都进行校验，减少了"跳步出错"的概率。
